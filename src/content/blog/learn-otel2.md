前回は OpenTelemetry の導入構想を立てましたが、今回は実装に向けて検証を進めました。複数プロジェクトからのデータを効果的に管理するためのアプローチがわかってきた。

## 実装の課題：複数プロジェクトのデータを見分ける

一元的に Collector を設置してデータを集約することは良いが、問題があった。

**いろんなところから入ってくるデータをどうやって見分けるのか？**

各プロジェクトから送られてくるトレース、ログ、メトリクスが混在する状態では、「このデータはどのプロジェクトからのものなのか？」を特定するのが難しい。

## 解決策：リソース属性で識別子を付与

送信元（各プロジェクト）でリソース属性に識別子を付与することで解決可能。

環境変数で設定する例：
```bash
OTEL_RESOURCE_ATTRIBUTES="gcp.project.id=pj-frontend-stage,environment=staging"
```

この方法なら、コード側である程度制御でき、API キーで上書きするような複雑な設定不要で、運用的にも分かりやすいかも！

## 可視化時の検索方法

リソース属性を活用すれば、GCP の各ツールで効率的にプロジェクト別にデータを絞り込める：

### ① Cloud Trace（トレース）
検索窓に `gcp.project.id=pj-backend-prod` と入力するだけで、そのプロジェクトのトレースだけを瞬時に絞り込める。

### ② Cloud Logging（ログ）
ログエクスプローラで以下のクエリを使用：
```
jsonPayload.resource.attributes."gcp.project.id" = "pj-backend-prod"
```

### ③ Cloud Monitoring（メトリクス）
Metrics Explorer 画面で「Group By（グループ化）」や「Filter（フィルタ）」の条件に `gcp.project.id` を指定することで、プロジェクト別のグラフを並べて比較したり、特定のプロジェクトの負荷だけをダッシュボードに表示させたりできる。

## 学び

* 複数プロジェクトのデータを一元管理するには、**リソース属性で識別子を付与する**のが効果的
* GCP の各可視化ツール（Trace、Logging、Monitoring）は識別子を使った検索・フィルタリングに対応している
* コード側で制御できる設定は、API キー上書きより運用的に分かりやすくて良い
